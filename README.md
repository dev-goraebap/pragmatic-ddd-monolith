# backend-pragmatism-design

> OOP + DI 백엔드를 위한 구조-first 설계 스킬.
> "이 코드 어디에 둬야 하는가"를 세 물리 레이어와 단방향 참조 그래프로 답합니다.
> 프레임워크 중립 (NestJS, Spring Boot, ASP.NET Core, Micronaut, Quarkus 등).

## 설치

```bash
npx skills add dev-goraebap/backend-pragmatism-design
```

## 무엇을 하는가

`app → module → shared` 세 레이어의 단방향 참조 그래프를 기준으로 백엔드 코드의 물리적 배치를 결정합니다.

프론트엔드의 FSD(Feature-Sliced Design)가 "레이어 계층 + import 규칙"만 핵심으로 두는 것처럼, 이 스킬은 백엔드에서 구조 그 자체에 집중합니다. 테스트 작성법이나 에러 분류 같은 디테일은 프로젝트마다 다르기 때문에 강제하지 않고, PROJECT.md(프로젝트 컨벤션)로 넘깁니다.

판단의 대부분은 [SKILL.md](./SKILL.md) 하나로 끝납니다. 더 깊은 설명이 필요할 때만 [reference/](./reference/)를 펼칩니다.

## 자기 규율: 원리 vs 실행 vs 컨벤션

이 스킬은 모든 진술을 셋 중 하나로 구분합니다.

- **A. 설계 원리** — 프레임워크와 무관하게 "왜"가 성립하는 것. SKILL.md와 reference 본문에 둡니다.
- **B. 프레임워크 실행 방식** — 같은 원리를 NestJS/Spring/.NET이 다르게 구현하는 것. 각 reference 하단 "프레임워크 노트"에 둡니다.
- **C. 팀 컨벤션** — 같은 프레임워크 안에서도 팀마다 갈리는 것. PROJECT.md로 외부화합니다.

# 철학: 왜 이렇게 설계했는가

## 1. 지향점 — 실용적이고 단순한 레이어드

이 스킬이 지향하는 구조는 레이어드 아키텍처를 기반으로 DDD 기법을 일부 차용한 실용적 아키텍처입니다. 헥사고날·클린·완전 CQRS의 도그마를 엄격히 따르지는 않습니다. 개념적 레이어 흐름은 전통적 방향(`Presentation → Application → Domain ← Infrastructure`)을 유지합니다. 진짜 과제는 그다음입니다 — 이 개념적 레이어를 물리적 폴더 구조에 어떻게 효율적으로 투영하느냐.

개념적 구조를 그대로 프로젝트에 옮기면 트레이드오프가 따라옵니다. 순수 레이어드는 기능 응집과 확장에 불리하고, 모듈식 헥사고날은 모놀리식에 불필요한 복잡도를 더합니다. 이 스킬은 그 사이에서 모놀리식에 맞는 타협점을 찾습니다.

## 2. 기존 두 물리 구조의 한계

전통적인 폴더 구성은 크게 둘인데, 모놀리식에서는 각각 뚜렷한 한계가 있습니다.

- **레이어-first** (`presentation/`, `application/`, `domain/` 아래에 BC) — 개별 기능(BC) 단위의 응집도가 무너집니다. `org` 하나를 고치려면 여러 레이어 폴더를 동시에 뒤져야 합니다.
- **BC-first** (`org/` 아래에 자체 레이어) — BC 간 참조가 일어나는 오케스트레이션에서 딜레마에 빠집니다. 격리 수단이 둘 있지만 모두 비쌉니다.
  - 인터페이스 + DIP 격리: 단일 모놀리식에서 매번 포트·어댑터를 설계하는 것은, "언젠가 마이크로서비스로 쪼갤지 모른다"는 막연한 가정에 매일 보험료를 내는 것과 같습니다.
  - 도메인 이벤트 비동기 격리: 진짜 비동기 요구에는 적합하지만, 즉각적 동기 트랜잭션이 필요한 일까지 이벤트로 강제하면 트랜잭션과 디버깅 난도가 폭발합니다.

두 격리 모두 "한 BC의 Application Service가 다른 BC의 도메인을 써야 한다"는 현실에서 출발합니다. 그런데 모놀리식에서는 격리가 주는 이점보다 그 비용이 훨씬 큽니다.

## 3. 대안 — 물리적 참조 방향과 변동성에 따른 재분류

'BC가 먼저냐 레이어가 먼저냐'의 양자택일에서 벗어나, 물리적 참조 방향의 제어 가능성과 변동성 주기를 기준으로 다시 나눕니다.

- 도메인은 BC 단위로 응집합니다. 같은 BC의 도메인 요소는 비즈니스 규칙을 공유하며 함께 변하고 함께 검증됩니다. 도메인은 외부 화면이나 인프라 결정과 무관하게 독립적으로 설계되고, 의존성 0의 Solitary Unit Test로 검증될 수 있어야 합니다.
- 영속성 인프라는 도메인의 동반자입니다. Repository 구현은 자신이 다루는 도메인 엔티티와 1:1로 강하게 결합합니다. 그래서 도메인과 그 영속성을 같은 BC 폴더(`module/<bc>/`)에 함께 둡니다.
- 애플리케이션 레이어는 BC 밖으로 끌어올립니다. 애플리케이션 서비스는 여러 BC를 교차 조회·통제하는 결합의 진원지이고, 동시에 컨트롤러와 라이프사이클·DTO를 공유합니다. 이를 특정 BC 안에 두면 BC 간 순환 참조가 쉽게 생깁니다. 그래서 프레젠테이션과 통합해 `app/`이라는 최상위로 격상합니다.

`app/`은 하위의 모든 `module/`을 자유롭게 참조할 수 있는 단일 결합 지점입니다. 무리한 인터페이스나 이벤트 우회 없이, 필요한 모듈을 주입받아 유스케이스를 조립합니다. 컨트롤러·서비스·DTO가 한 기능 폴더에 모이므로 프런트엔드 스펙 변경에도 유연합니다.

### 등가교환

이 선택에는 대가가 있습니다. 오케스트레이션 서비스가 도메인이 아니라 기능(feature) 구조에 종속된다는 점입니다. 대신 기능을 사용자/액터 관점의 평면 슬라이스로 정돈하고, 채널(REST·크론·컨슈머)을 슬라이스 안의 프래그먼트로 내립니다. 그러면 채널 분할에서 오던 중복이 구조적으로 사라집니다. (자세히는 SKILL.md)

## 4. 테스트로 본 인터페이스 기준

인터페이스의 목적은 헥사고날 형식을 맞추는 것이 아니라, 테스트에서 테스트 더블을 교체 주입할 슬롯을 제공하는 것입니다.

- Repository는 인메모리 더블로 상태 변화를 검증할 가치가 매우 높으므로 추상으로 노출합니다. (TypeScript의 `interface`는 빌드 시 사라져 DI 토큰으로 식별되지 않으니 `abstract class`를 씁니다 — 프레임워크 실행 디테일.)
- 외부 서드파티 연동은 '요청이 올바로 전달됐는가'의 행위 검증으로 충분한 경우가 많으므로, 기본은 구체 클래스로 두고 모킹으로 검증합니다.

## 5. 얕은 CQRS

화면 조회(Read)와 비즈니스 도메인(Write)은 생명주기가 근본적으로 다릅니다. Write는 애그리게이트 단위로 일관성을 엄격히 검증하고, Read는 여러 테이블을 조인한 평면적 데이터를 화면 규격에 맞춰 반환합니다. 도메인 엔티티를 화면 스펙에 맞춰 가공하려 들면 N+1, 불필요한 노출, 무거운 변환 코드가 생깁니다.

조회 서비스를 'DB를 본다'는 이유로 `module/<bc>/infrastructure/`에 두면 문제가 생깁니다. 반환 타입(Response DTO)이 상위 `app/`에 속하므로 역방향 의존이 발생하기 때문입니다. 그래서 조회 인프라(Query Service)를 `app/<feature>/`에 함께 둡니다. "인프라는 한 폴더에만 모여야 한다"는 형식적 도그마를 깨는 대신, 변환 보일러플레이트와 우회 참조를 없앱니다.

## 6. 실질적 이점

1. 순환 참조의 원천 차단 — `app → module → shared` 단방향.
2. 인터페이스 역할 재정립 — 테스트 더블이 필요한 경계(Repository, 주요 외부 연동)에만 선택적으로 추상화.
3. 폴더 구조가 곧 테스트 가이드 — `module/<bc>/domain/`은 Solitary, `app/<feature>/*.service`는 Sociable, 영속성·조회는 Integration(선택).
4. 도메인 보존 — 변동성 큰 화면(app)과 정적인 규칙(module)을 물리적으로 격리.
5. 학습 곡선 최소화 — "불변 규칙이면 `module`, 화면·외부 결합이면 `app`."

## 7. 구조적 유연성 (자유도)

아키텍처는 고정된 법률이 아니라 상황에 적응하는 가이드입니다. 특히 `app/` 안에서 슬라이스 간 공통 로직을 어떻게 다룰지는 단일 지침으로 강제하지 않습니다. 먼저 "이거 도메인 아냐? → `module`로 내려라"를 묻고, 그래도 남는 순수 코드 중복은 팀의 기술적 판단(OOP 합성 등)에 맡깁니다. 형식 규격을 사수하려고 더 복잡한 우회 코드를 양산하지는 않습니다.

## 8. 이 아키텍처가 맞지 않는 경우

이 구조는 "한 팀(또는 소수 팀)이 여러 BC를 함께 다루는 단일 DB 모놀리식"의 중간 규모대에 맞춰져 있습니다. 그 바깥에서는 과하거나 부족합니다.

- **너무 단순한 프로젝트 (과함)** — `app` 한 레이어로 끝나고 슬라이스 간·BC 간 참조가 사실상 없는 경우입니다. 3레이어 분할과 `module`/`shared` 경계가 이득보다 오버헤드가 큽니다. 단순 CRUD, 프로토타입, 소규모 내부 도구라면 단일 레이어로 충분합니다.
- **팀=모듈 소유의 대형 멀티팀 (부족함)** — 팀 경계가 곧 모듈 경계이고 독립 소유권·독립 배포가 중요한 경우입니다. 이 구조의 핵심인 "`app`이 모든 `module`을 자유롭게 참조한다"가 오히려 팀 간 결합을 만듭니다. 이럴 땐 BC 간 강한 격리(헥사고날의 명시적 포트, 모듈러 모놀리스, 마이크로서비스)가 더 적합합니다. 2장에서 "보험료"라 불렀던 격리 비용이 여기서는 실제로 값을 합니다.
- **진짜 이벤트 기반/비동기 도메인** — 이벤트 소싱이나 완전한 CQRS가 본질인 시스템입니다. 얕은 CQRS로는 부족합니다.
- **전제 밖 환경** — 프론트엔드, DI 없는 raw Express/Fastify, Go식 명시 와이어링, 순수 함수형, 그리고 마이크로서비스 분할 그 자체. (SKILL.md 전제 참조)

> 한 줄 기준: BC 간 참조(오케스트레이션)가 거의 없으면 과하고, 팀 단위로 강하게 격리해야 하면 부족합니다. 그 사이가 스위트 스팟입니다.
