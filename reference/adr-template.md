# ADR 템플릿 — 결정을 기록하는 법

이 스킬은 **원칙까지만** 정한다. 구체 구조·명명·결합 방식은 팀이 정하고, **그 결정과 근거를 ADR(Architecture Decision Record)로 남긴다.** 나중에 "왜 이렇게 했더라"를 코드가 아니라 결정 기록으로 답할 수 있게 하기 위함이다.

## 쓰는 법

- ADR 하나 = **결정 하나**. 짧게, 결정이 내려질 때 바로.
- 저장 위치는 자유 — 보통 저장소의 `docs/adr/` 아래 `NNNN-제목.md`(예: `0001-orm-annotation-policy.md`)로 번호를 매겨 시간순으로 쌓는다.
- 결정이 바뀌면 기존 ADR을 지우지 말고 **새 ADR로 대체(Supersedes)** 하고 옛 ADR의 상태를 `Superseded`로 표시한다.

## 템플릿

```markdown
# NNNN. <결정을 한 문장으로>

- 상태: 제안 / 채택 / 폐기 / 대체됨(→ NNNN)
- 날짜: YYYY-MM-DD
- 관련: <관계된 컨텍스트 / 다른 ADR 번호>

## 맥락
무엇이 문제이고 왜 지금 결정해야 하는가. 제약(단일 배포·단일 DB, 팀 사정, 일정 등).

## 결정
무엇을 하기로 했는가. 명확하고 단정적으로.

## 근거
왜 이 선택인가. 어떤 원칙·트레이드오프를 따랐는가.

## 대안
검토했으나 택하지 않은 선택지와 그 이유.

## 결과
이 결정으로 생기는 이득과 비용, 후속으로 지켜야 할 것.
```

## 이 아키텍처에서 ADR로 남길 만한 결정

- **명명·폴더 구조** — 패키지/레이어 명명 규칙, 컨텍스트별 폴더 구조.
- **컨텍스트 경계** — 무엇을 한 컨텍스트로 묶고 어디서 가르는가, 컨텍스트 목록.
- **결합 방식** — 관계별 Conformist vs ACL 선택과 이유.
- **ORM 애노테이션 정책** — 도메인 객체에 직접 사용할지(순수성과의 타협 여부).
- **공개 계약** — 계약 구현 방식(기존 서비스 통합 / 외부용 분리), 가시성 강제 방식(리뷰 / 모듈 시스템 / 아키텍처 테스트 등).
- **읽기 모델 전략** — 프로젝션 방식, 크로스 컨텍스트 직접 조회 허용 범위.
- **공통/유틸 경계** — 도메인 무관 영역에 특정 컨텍스트가 새어 들지 않게 하는 규칙.
