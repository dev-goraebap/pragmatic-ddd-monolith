# decoupling — 결합 푸는 방법

> 컨텍스트 간 참조를 어떻게 다스릴지. 실제로 풀 때 펼친다.
> 배경: [백엔드 실용주의 디자인](https://goraebap.xyz/posts/backend-pragmatism-design/).

## 먼저: 참조하는 이유 3가지

컨텍스트 밖을 참조하는 이유는 대개 셋이다. 이유에 따라 방법이 갈린다.

1. 다른 컨텍스트의 **데이터를 조회** → §읽기 모델(얕은 CQRS).
2. 다른 컨텍스트 기능이 **한 트랜잭션으로 함께 보장** → 폴더 레이어 / DIP / 이벤트.
3. **기술적 외부 모듈**(이메일·푸시·스토리지) 사용 → 보통 그냥 가져다 씀(인터페이스 없이 구체 클래스 + 테스트는 mock).

> 같은 컨텍스트 *안*의 모듈끼리는 굳이 아래 방법을 쓰지 않는다 — 자유롭게 참조한다.

## 방법 A. 폴더 레이어 2분할

결합을 일으키는 오케스트레이션을 별도 레이어로 내린다. **참조당하는 쪽을 더 하위 레이어로.**

- **상위 레이어**: 도메인 모델을 가져다 쓰는 컨트롤러·오케스트레이션 서비스.
- **하위 레이어**: 도메인 모델 + 인프라스트럭처.

규칙: ① 상위 → 하위 참조 가능. ② **같은 레이어 모듈끼리는 직접 참조 금지.** ③ 그래도 남는 예외는 팀 합의.

```
상위 (controller + orchestration)   content / iam ...
        │  (공평하게 하위를 가져다 씀)
        ▼
하위 (domain + infra)               content-domain / iam-domain ...
   (같은 레이어끼리 ❌)
```

- 하위의 도메인·리포지토리를 상위 여러 모듈이 공평하게 쓴다. 같은 사용 패턴이 반복되면 그때 더 추상적인 서비스로 묶는다.
- 필요할 때 하나씩 내려도, 일관성을 위해 한 번에 분리해도 된다.
- 트레이드오프: 상위가 많아지며 하위에 불필요한 기능이 끼는 *오염* 우려 → 하위에서 한 번 더 캡슐화로 조절 가능.
- 언제: 중규모 이상, 결합이 늘 때. (컨텍스트 2~3개·단방향이면 과하다 → 그냥 둬라.)

## 방법 B. DIP 인터페이스

모듈을 한 레이어에서 관리하면서 **내부 리포지토리·엔티티를 직접 노출하지 않는다.** 외부가 필요한 만큼만 인터페이스로 드러내고, 순환 조짐이 보이면 **DIP로 방향을 단방향으로 되돌린다.**

```
content 모듈 ──필요한 기능만 호출──▶ 《interface》 Notifier
notification 모듈 ┄┄구현┄┄▶ Notifier
```

- 헥사고날과 닮았다. 자주 참조당하는 모듈은 직접 기능을 공유하거나 **공유 커널(Shared Kernel)** 을 둘 수도 있다.
- 트레이드오프: 캡슐화·순환 차단을 얻지만, 인터페이스를 만들고 유지하는 비용이 든다.
- 언제: 한 레이어 구조를 유지하며 캡슐화하거나 순환을 끊고 싶을 때.

## 방법 C. 동기 인-프로세스 이벤트

참조를 *허용하되 다스리는* A·B와 달리, **참조 자체를 끊는다.** 발행 모듈은 "사실"만 알리고 손을 뗀다.

```
PostService ──PostPublished 발행──▶ 인-프로세스 이벤트 버스 ──동기──▶ 알림 핸들러 ──▶ NotificationRepository
```

- **모놀리식 안에서 동기로** 처리되는 인-프로세스 이벤트다(메시지 브로커 분산 비동기 아님). 핸들러가 **같은 트랜잭션 안에서 함께 실행**되므로 "게시물 생성 + 알림 이력 생성"의 일관성을 그대로 지킨다.
- **실제 외부 발송(이메일·푸시)은 tx 밖**으로 뺀다(이력은 tx 안, 발송은 커밋 후).
- 트레이드오프: 결합이 느슨해지는 대신 **흐름이 코드에서 한눈에 안 보이고 디버깅이 간접적**이 된다. 규모가 커질수록 이점이 크다.
- 언제: 부수효과·알림처럼 호출 방향을 끊고 싶을 때.

## 얕은 CQRS — 읽기 모델

진짜 복잡도를 키우는 건 **조회**다. 쓰기는 경계·규칙을 엄격히 지키되, 조회는 두 갈래로 푼다.

### 1) 화면용 데이터 → 전용 쿼리 서비스
목록 화면에 글 제목·작성자 이름·댓글 수·카테고리가 한꺼번에 필요한 경우. 쓰기 모델을 거치지 말고 **화면이 필요한 모양 그대로 가져오는 쿼리 서비스**를 둔다. **컨텍스트를 가로질러 조인하거나 DB 뷰를 써도 된다** — 쓰기를 안 건드리는 순수 읽기라 경계 규칙을 강요할 이유가 없다.

### 2) 비즈니스 흐름용 조회 → 내 언어의 읽기 전용 모델
다른 컨텍스트 데이터를 *검증*하거나 흐름의 입력으로 쓸 때. (동기 이벤트는 *행위를 호출*하는 것이라 조회를 대신 풀어주지 않는다.)

> 데이터를 **생성·변경하는 주체는 한 컨텍스트가** 관리하되, 그 데이터가 다른 흐름에서 다른 의미로 쓰인다면 **내 컨텍스트 언어로 된 읽기 전용 모델**을 만들어 자기 리포지토리로 같은 테이블을 조회한다.

예: 알림 컨텍스트가 원하는 건 '사용자'가 아니라 **수신자(`Recipient`)** 다. IAM의 user 테이블을 알림 컨텍스트의 `Recipient` 읽기 전용 모델로 조회한다. 무거운 CQRS 인프라(별도 저장소·이벤트 소싱) 없이 단일 DB 위에서 경계를 지키며 조회 비용을 낮춘다.

## 선택 가이드 (요약)

| 상황 | 권장 |
| --- | --- |
| 컨텍스트 2~3개, 참조 단방향 | 그냥 둔다 |
| 중규모↑, 결합 증가 | 폴더 레이어 2분할 |
| 한 레이어 유지 + 캡슐화/순환 차단 | DIP 인터페이스 |
| 부수효과·알림 등 방향 끊기 | 동기 이벤트 |
| 화면 조회 | 전용 쿼리 서비스(경계 가로질러 OK) |
| 흐름 내 타 컨텍스트 조회 | 내 언어 읽기 전용 모델 |

## 프레임워크 노트

- **동기 이벤트 버스**: NestJS `@nestjs/event-emitter`(EventEmitter2, 기본 동기) · Spring `ApplicationEventPublisher` + `@EventListener`(또는 `@TransactionalEventListener`) · .NET MediatR in-process notifications.
- **트랜잭션 전파**(이벤트 핸들러가 같은 tx): NestJS `AsyncLocalStorage`(`nestjs-cls`) · Spring `@Transactional` 전파 · .NET `TransactionScope`/scoped `DbContext`.
- **읽기 전용 모델/뷰**: TypeORM `@ViewEntity` 또는 쿼리빌더 `getRawMany`→DTO · JPA DTO projection 또는 `@Immutable`+`@Subselect` · EF Core keyless `ToView`/`ToSqlQuery`.
- **외부 어댑터**(이메일·푸시·스토리지): 인터페이스 없이 구체 클래스로 두고 테스트에서 mock. (단 Repository는 추상 유지 — 인메모리 테스트 더블 슬롯)
