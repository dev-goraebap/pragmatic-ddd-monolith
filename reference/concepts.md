# concepts — 빌려온 DDD 어휘

> DDD에서 "경계를 긋는 언어"만 실용적으로 빌려온다. 정의가 필요할 때 펼친다.
> 배경: [경계를 긋는 언어, 도메인 주도 설계](https://goraebap.xyz/posts/language-of-boundaries/).

## 도메인 / 하위 도메인

- **도메인** = 우리가 풀려는 문제의 영역(= 프로젝트 그 자체). 관점에 따라 달라진다(나에겐 곁다리 '결제'가, PG사에겐 도메인 전체).
- **하위 도메인(subdomain)** = 도메인을 세분화한 영역. 우리가 "주문 도메인"이라 부르는 건 사실 하위 도메인이다.

세 분류(느슨하게 적용):

| 하위 도메인 | 성격 | 구현 |
| --- | --- | --- |
| **핵심(Core)** | 경쟁 우위·복잡·변동 잦음 | 직접 개발 |
| **일반(Generic)** | 검증된 해법 존재(인증·결제·파일) | 구매/도입 |
| **지원(Support)** | 단순하지만 핵심에 필요(공지·기본정보) | 직접/하청 |

> 작은 프로젝트에서 "경쟁 우위" 기준이 애매하면 느슨하게: **이름값이 크거나, 복잡도가 높거나, 다른 영역을 끌어다 흐름을 주도하면 핵심.** 핵심은 여러 개일 수 있다.

## 바운디드 컨텍스트

- **모델과 언어가 통용되는 개념적 경계.** 같은 단어가 영역마다 다른 의미를 가지는 지점이 경계다. (카탈로그의 '상품'=전시 정보, 주문의 '상품'=구매 시점 스냅샷, 재고의 '상품'=수량)
- 경계 안의 일관된 언어 = **유비쿼터스 언어**.
- **MSA의 서비스 경계가 아니다.** 모놀리식 안에서도 경계를 긋는다. 물리적 분리 지시가 아니라 *모델을 나누는 개념적 경계*.
- **앱 아키텍처 ≠ 데이터 아키텍처.** 코드에서 컨텍스트별로 '상품'을 따로 정의해도, 데이터는 하나의 `product` 테이블에서 출발해도 된다.
- 하위 도메인(문제 공간) ↔ 컨텍스트(해결 공간)가 항상 1:1은 아니다. 모놀리식에선 대체로 일치시키는 편이 관리하기 편하다.
- 모놀리식에선 경계가 물리적으로 강제되지 않으므로 **의식적으로 그어야 하는 선**이다.
- **컨텍스트 ≠ 크기.** 모듈(애그리거트)이 하나뿐이어도 컨텍스트다(도메인 모델은 애그리거트로 *이뤄지는* 것이지 여러 개여야 성립하는 게 아니다). 잣대는 개수가 아니라 **자기만의 언어와 책임**. 작을 땐 '컨텍스트냐 기능이냐' 못 박지 말고 작게 두고 키운다.
- **경계는 양방향으로 변한다.** 나눴던 모듈이 다시 묶이기도, 하나로 본 컨텍스트가 복잡도·팀 작업 방식에 따라 다시 쪼개지기도 한다.

## 도메인 모델

현실 도메인을 코드 객체로 옮긴 것. **엔티티는 도메인 모델의 한 조각일 뿐**(`엔티티 ⊂ 도메인 모델`).

| 구성 요소 | 한 줄 정의 |
| --- | --- |
| **엔티티(Entity)** | 식별자로 구분되고 생명주기를 갖는 객체 |
| **값 객체(Value Object)** | 식별자 없이 값 자체로 동등성이 결정되는 객체 |
| **애그리거트(Aggregate)** | 함께 일관성을 지켜야 하는 엔티티·값 객체의 묶음 |
| **도메인 서비스** | 특정 엔티티에 넣기 애매한 도메인 로직 |

- **ORM `@Entity`는 영속성 모델**("어떻게 저장되나")이고, DDD 엔티티는 도메인 개념("어떤 규칙을 갖나")이다. 모놀리식이면 한 클래스가 둘을 겸해도 되지만, 그러려면 **행동·규칙을 담아야** 한다(데이터 그릇 금지).
- **빈약(anemic)**: 규칙이 서비스에 흩어짐 / **풍부(rich)**: 규칙이 모델 안에서 지켜짐. 핵심일수록 rich의 보상이 크고, 단순 CRUD는 anemic도 괜찮다.
- **불변식(invariant)**: 어떤 상태에서도 깨지면 안 되는 규칙. rich 모델은 객체가 스스로 지킨다.
- **VO는 정말 필요할 때만.** 일반적인 값(이메일·금액)은 검증 라이브러리로 충분. 우리 도메인만의 특별한 정책이 얹힐 때 만든다.

## 애그리거트 경계

- **애그리거트 루트** = 외부에서 닿는 유일한 입구. 묶음 안 객체는 루트를 거쳐서만 접근.
- **애그리거트끼리는 객체 참조가 아니라 ID로 참조** (`Post`는 `authorId`만, `Comment`는 `postId`만).
- 경계 기준은 *"함께 사라지는가(cascade)"가 아니다.* → **"하나의 불변식을 함께 지켜야 하는가, 외부가 반드시 루트를 거쳐야 하는가."** (댓글은 게시글 없이도 독립적으로 다뤄지니 별개 애그리거트)
- **애그리거트 경계 ≠ 컨텍스트 경계** (다른 층위). 한 컨텍스트 안에 여러 애그리거트가 산다(`Post`·`Comment`는 같은 '콘텐츠' 컨텍스트의 다른 애그리거트). 한 애그리거트가 여러 컨텍스트에 걸치는 일은 없다.

## 트랜잭션 실용

- **"1 트랜잭션 = 1 애그리거트"는 분산(MSA, 컨텍스트별 저장소 분리) 전제**에서 나온 규칙이다.
- **모놀리식 단일 DB에선 여러 애그리거트를 한 tx로 묶어도 된다** (게시글 생성 + 알림 이력 생성). 단 **실제 외부 발송(이메일·푸시)은 tx로 묶을 단위가 아니다.**
