# critique

> 책임·의존성·복잡도를 종합적으로 리뷰. 점수와 권고를 함께 제공한다. 카테고리: Evaluate.

## 사용 시점

- 리뷰 전 self-review.
- 큰 PR 머지 직전.
- 분기/스프린트 회고 시 코드 건강 점검.

## 입력

- 리뷰 대상(파일, 디렉터리, BC, 또는 PR diff).

## 절차

1. (TBD) **책임 분배 점검**: Controller(0 로직) → Service(오케스트레이션) → Domain(결정) 흐름이 맞는지.
2. (TBD) **의존성 흐름 검증**: `app → module → shared` 위반 없는지.
3. (TBD) **추상화 정합성**: 인터페이스가 테스트 더블 슬롯 목적에 정합한가, 죽은 추상은 없는가.
4. (TBD) **CQRS 분리**: Query Service가 CUD에 끼어들지 않는가.
5. (TBD) **테스트 정합성**: 도메인은 Solitary, 유스케이스는 Sociable, 통합은 선택인가.
6. (TBD) **에러 분류 정합성**.
7. (TBD) 항목별 1~5점 + P0~P3 액션 plan.

## 출력

- 점수표, 항목별 코멘트, 우선순위가 매겨진 액션 plan.

## 관련

- [[audit]] - 기계적·체크리스트형 검사 (보완 관계).
- [[refactor]] - 권고를 코드로 반영.

## 프레임워크 노트

본 명령은 코드 리뷰. 프레임워크 무관하나 각 프레임워크의 관용 패턴(예: NestJS `@Module` 의존 그래프, Spring `@Component` 스캐닝 범위)을 추가 휴리스틱으로 활용 가능.
